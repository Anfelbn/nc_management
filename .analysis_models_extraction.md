- **`_compute_can_access_fac`** (`@api.depends('fac_ids', 'fac_ids.responsable_id', 'fac_ids.current_handler_uid')`) : True si l'utilisateur est RMQSE (`group_responsable_qualite`), OU s'il est `responsable_id` ou `current_handler_uid` de la première FAC liée.

### Contraintes (`@api.constrains`)

- **`_check_fnc_required`** : sur `direction_id`, `description`, et tous les `type_*`. Bypass si `context.get('skip_fnc_validation')` ou `name == 'New'`. Exige : `direction_id` renseigné, au moins un type NC sélectionné, `description` non vide.
- **`_check_fonction_visa_requirements`** : sur `fonction_visa`. Si renseigné (et `name != 'New'`), exige : au moins un type NC, `description` non vide, `signale_par_id`, `date_signalement`.

- **`_onchange_fonction_visa`** : valide les pré-requis (type NC, description, signalement, date) avant d'autoriser `fonction_visa` ; sinon reset + warning. Si tout OK et `state == 'draft'` → passe à `'submitted'`.
- **`_onchange_autofill_fonction_visa`** (multi-dépendances : tous les `type_*`, `description`, `signale_par_id`, `date_signalement`) : si `fonction_visa` vide et toutes conditions réunies, auto-remplit avec `employee.visa_no or employee.job_id.name or employee.name` de l'utilisateur courant.
- **`_onchange_responsable_action_id`** : exige `realise_par_id` et `date_realisation` avant d'autoriser `responsable_action_id`, sinon reset + warning.
- **`_onchange_autofill_responsable_action`** (depends `realise_par_id`, `date_realisation`) : auto-remplit `responsable_action_id` avec l'employé courant si conditions réunies.
- **`_onchange_autofill_signature`** (depends `superieur_id`, `date_validation`, `signature`) : si `state == 'in_progress'`, auto-remplit `superieur_id`, `date_validation` (today), `signature` (visa/job/name) de l'employé courant.
- **`_onchange_traitement_complet`** (depends impact/action_immediate/analyse_causes/realise_par_id/date_realisation/trait_*) : si `state == 'submitted'` et traitement complet (`_traitement_complet()` ne renvoie rien manquant), auto-assigne `assigned_to_id` = employé courant.
- **`_onchange_assigned_to_id`** : si `state` dans (`submitted`,`in_progress`), vérifie via `_traitement_complet()` que tous les champs de traitement sont remplis, sinon reset `assigned_to_id` + warning listant les champs manquants.
- **`_onchange_signature`** : si `signature` renseignée et `state == 'in_progress'` → passe à `'validated'`.
- **`_onchange_direction_id`**, **`_onchange_department_id`**, **`_onchange_service_id`**, **`_onchange_section_id`** : cascade de réinitialisation des champs hiérarchiques enfants (direction → reset department/service/section/equipe ; department → reset service/section/equipe ; etc.)
- **`_onchange_type_exclusive`** (multi-dépendances sur tous les `type_*`) : sélection exclusive — un seul type NC peut être actif à la fois. Si `name == 'New'`, bloque la sélection avec un warning ("Numéro FNC requis"). Sinon désactive tous les autres types quand un nouveau est cochée.

### Méthodes métier

- **`default_get`** (`@api.model`) : pré-remplit `signale_par_id` = employé de l'utilisateur courant, et remonte la hiérarchie du département de l'employé pour pré-remplir `direction_id`, `department_id`, `service_id` selon `scaek_level`.
- **`create`** (`@api.model`) : simple passthrough à `super()`.
- **`write`** (`@api.multi`) — méthode la plus complexe du modèle :
  1. Empêche la modification de `name` une fois différent de `'New'` (ValidationError).
  2. Transitions d'état automatiques : si `fonction_visa` renseigné et `state == 'draft'` → `'submitted'` ; si `signature` renseignée et `state == 'in_progress'` → `'validated'`.
  3. "Smart sync" : mémorise les anciennes valeurs de `description`, `analyse_causes`, `direction_id` avant écriture.
  4. Après `super().write()`, propage les changements vers TOUTES les FAC liées (`fac_ids`) — synchronise `rappel_nc` (depuis `description`), `analyse_causes`, `direction_id` — uniquement si la FAC n'a pas déjà une valeur différente de l'ancienne (évite d'écraser une modification manuelle sur la FAC).
  5. **Création automatique de FAC** : si tous les champs de traitement (`treatment_fields` = impact, action_immediate, analyse_causes, trait_*, assigned_to_id) sont renseignés et qu'il n'y a pas encore de FAC et que `state` ∈ (`submitted`,`in_progress`) et `assigned_to_id` défini et qu'au moins un type de traitement est cochée :
     - Si `state == 'submitted'` → passe à `'in_progress'` avec `date_in_progress = today`.
     - Crée une `nc_management.corrective_action` (en `sudo()` du `responsable_id`) avec `fnc_id`, `ref_document`, `direction_id`, `rappel_nc`, `analyse_causes`, `responsable_id` (= user de l'employé assigné), `responsable_analyse_id` (= employé assigné).
  6. Si `assigned_to_id` change, synchronise `responsable_id` (user lié) et `responsable_analyse_id` (employé) sur toutes les FAC liées.
- **`unlink`** (`@api.multi`) : seul un `group_responsable_qualite` peut supprimer les FNC d'autres créateurs (sinon UserError).
- **`_traitement_complet`** : retourne une liste de libellés des champs de traitement manquants (type de traitement, action immédiate, réalisé par, date de réalisation, analyse des causes, impact).
- **`_is_creator`** : `submitted_by_id == env.user`.
- **`_is_assigned`** : l'employé lié à l'utilisateur courant est l'`assigned_to_id`.

### Actions / boutons (`@api.multi`)

- **`action_valider_fnc`** : passe `state='validated'`, auto-remplit `superieur_id` (employé courant si vide), `date_validation` (today si vide), `signature` (visa/job/name si vide). Retourne une action de réouverture du formulaire en mode lecture (`form_view_initial_mode: view`).
- **`action_open_send_wizard`** : ouvre le wizard `nc_management.send_fnc_wizard` (target='new'). Lève une UserError si `name == 'New'` (numéro non généré).
- **`action_open_number_wizard`** : ouvre le wizard `nc_management.number_generator_wizard` (target='new'). Lève une UserError si `name != 'New'` (déjà assigné).

- **`_onchange_fnc_id`** : auto-remplit `ref_document`, `rappel_nc` (← `description` FNC), `analyse_causes`, `direction_id`, `date_fnc` (← `date` FNC), `responsable_analyse_id` (← `assigned_to_id` FNC), `responsable_id` (← `user_id` de l'employé assigné).

- **`_state_from_visas(visa_analyse, visa_actions, qse_visa, visa_cloture)`** : fonction pure qui dérive l'état depuis les visas — `closed` si `visa_cloture`, `validated` si `qse_visa`, `in_progress` si `visa_actions`, `submitted` si `visa_analyse`, sinon `draft`.

  - `visa_cloture` requiert : `cloture_par_id`, `date_cloture`.

- **`_check_visa_analyse`**, **`_check_visa_actions`**, **`_check_qse_visa`**, **`_check_visa_cloture`** (`@api.constrains`) : versions serveur (bloquantes) des mêmes validations que `_onchange_visa_state`, chacune levant une `ValidationError` listant les champs manquants.
- **`write`** (`@api.multi`) : pour chaque enregistrement, recalcule `state` via `_state_from_visas` en tenant compte des nouvelles valeurs de `vals` (force toujours l'état depuis les visas). Si nouvel état `'validated'` et `date_validated` vide → today. Si `'closed'` et `cloture_par_id`/`date_cloture` vides → auto-remplit (employé courant / today). Note : `break` après le premier enregistrement (logique conçue pour formulaire mono-enregistrement).

## 9. NcDashboard (`nc_management.dashboard`)

**Fichier** : models.py, lignes 2447-3713

Modèle "service" sans persistance de données propres — fournit uniquement des méthodes `@api.model` appelées par le frontend JS du tableau de bord. Très volumineux (~1270 lignes).

### Méthodes

#### `get_plan_smi_stats()`
Calcule les statistiques du tableau de bord SMI :
- **`get_categorie(field_name)`** (fonction interne) : pour un champ booléen donné (ex: `type_reclamation`), recherche les FNC correspondantes, puis les FAC liées, et calcule : `total`, `efficace`/`non_efficace` (via `actions_efficaces`), `realise_100`/`realise_50plus`/`realise_50moins` (via `state` de la FAC : `closed` / `in_progress`+`validated` / `draft`+`submitted`), `taux` (% d'efficacité).
- `categories` : liste pour 4 catégories — Réclamation PI, NC produit, Environnement, SST.
- `categories_chart` : 12 catégories basées sur `plan.nature` (mêmes valeurs que la sélection `nature` de `PlanActionSmi`), calculées sur les plans intégrés (`is_global=False`, `submission_state='integre'`) — `taux` = % de plans avec `efficacite=='oui'`.
- `processus` : 4 indicateurs de "Taux de clôture par processus" — Analyse et amélioration (FAC closes / total FAC), Process contrôle qualité (FNC type_nc_produit closes / total), Santé sécurité environnement (FNC type_sst OU type_environnement closes / total via `taux_multi`), Commercialisation/Réclamation (FNC type_reclamation closes / total).
- Fonctions internes `taux_proc(field_name)` et `taux_multi(fields_list)` calculent des taux de clôture (% de FNC `state=='closed'` parmi celles ayant le type donné).

#### `get_efficacite_categorie(field_name)`
Version simplifiée de `get_categorie` : pour un type de NC donné, retourne `total`, `efficace`, `non_efficace`, `taux` (sur les FAC liées via `actions_efficaces`).

#### `get_stats(period=None, target_year=None, target_month=None)`
**La méthode la plus volumineuse** (~370 lignes) — alimente le tableau de bord RMQSE (Responsable Management Qualité Sécurité Environnement). Logique :
- Détermine la période (`1m`/`6m`/`1y`, ou année/mois cible spécifique).
- Construit plusieurs domaines de recherche :
  - `fnc_period_domain` / `fac_period_domain` : FNC/FAC créées dans la période.
  - `received_fnc_period_domain` : FNC reçues par la RMQSE (envoyées par d'autres, `date_envoi` dans la période, `state != 'draft'`, `create_uid != uid`).
  - `fac_received_period_domain` : FAC correspondantes (liées aux FNC reçues OU FAC avec `date_envoi` dans la période créées par d'autres).
  - `own_fnc_domain` : FNC créées par la RMQSE elle-même.
  - `combined_fnc_domain` / `combined_fac_domain` : FNC créées par RMQSE OU de type audit (interne/externe) — pour les "Total FNC/FAC".
- **Compteurs FNC** (sur `combined_fnc_domain`) : `total_fnc`, `fnc_cours` (in_progress), `fnc_envoyes` (submitted), `fnc_closed`, `fnc_validated`, `fnc_brouillon` (draft), `taux_cloture`, `fnc_audit_interne`, `fnc_audit_externe`.
- **FNC en retard** (`fnc_retard_recs`) : FNC `state=='in_progress'`, `date <= today-7j`, créées par RMQSE — max 10.
- **Répartition par département** (`dept_list`) : top 6 départements par somme FNC+FAC reçues, avec pourcentages relatifs au max (`pct_fnc`, `pct_fac`) pour des barres de progression.
- **Répartition par type** (`type_counts`, `raw_type_counts`) : pourcentages de FNC reçues par type (nc_produit, reclamation, sst, environnement, audit, autres).
- **`_type_percentages(records)`** (fonction interne) : calcule la répartition % par type pour un ensemble d'enregistrements FNC.
- **`direction_stats`** : pour chaque direction (`hr.department` avec `scaek_level='direction'`), calcule `fnc_count`, `fac_count`, `fnc_types`, `fac_types`, `pct_fnc`, `pct_fac` (relatifs au max parmi les directions).
- **Compteurs FAC** (sur `combined_fac_domain`) : `total_fac`, `fac_open` (submitted), `fac_verif` (in_progress), `fac_closed`, `fac_brouillon`, `fac_validated`, `fac_efficace`, `taux_eff`, `taux_validation_fac`, `taux_cloture_fac`. Plus `fac_audit_interne`/`fac_audit_externe` (FAC liées à des FNC de type audit), et totaux combinés `audit_interne_total`/`audit_externe_total`.
- **FAC en retard** (`fac_retard_recs`, `fac_a_cloturer`) : FAC ouvertes (`submitted`/`in_progress`/`validated`) avec `date <= today-7j`. `fac_a_cloturer` classe chaque FAC par badge `red`/`orange`/`green` selon le nombre de jours ouverts.
- **`fac_recues_count`** : FAC reçues `state=='in_progress'` en attente d'approbation QSE.
- **Évolution mensuelle** (`monthly_fnc`, `monthly_fac`, `monthly_labels`) : pour chaque mois de la période, compte les FNC créées par la RMQSE (hors audits) et les FAC liées.

#### `get_user_stats(period=None, cal_year=None, cal_month=None)`
Alimente le tableau de bord "utilisateur standard" (non RMQSE) — ~240 lignes :
- KPI période : `fnc_total`, `fnc_brouillon`, `fnc_submitted`, `fnc_cours`, `fnc_validated`, `fnc_closed`, `fnc_taux_validation` ; idem pour FAC (`fac_total`, `fac_brouillon`, `fac_submitted`, `fac_cours`, `fac_validated`, `fac_closed`).

  - `user_fnc_retard_list` : FNC de l'utilisateur `state=='in_progress'`, `date_in_progress <= today-7j`.

#### `get_direction_details(direction_id, period=None)`
Pour une direction donnée, calcule les statistiques détaillées par département puis par service (drill-down hiérarchique) :
- `_pct(recs)` : répartition % par type NC pour un ensemble de FNC.
- Pour la direction : `fnc_count`, `fac_count`, `fnc_types`, `fac_types`.
- Pour chaque département (`scaek_level='departement'`, `parent_id=direction_id`) : mêmes stats + `services` (sous-liste, `scaek_level='service'`, `parent_id=dept.id`) avec leurs propres `fnc_count`/`fac_count`/`fnc_types`/`fac_types`.

#### `get_sender_info(model, record_id)`
Retourne les informations sur l'expéditeur d'un document (FNC, FAC, Plan SMI individuel, ou Plan d'Amélioration) pour affichage dans l'UI :
- `_split_name(full)` : sépare un nom complet en `(nom, prenom)` (premier mot / reste).
- `_fmt_dt(dt)` : formate une datetime en `'%d/%m/%Y à %H:%M'` (avec +1h, probablement correction de timezone).
- `_wizard_note(mod, rid)` : recherche dans `mail.message` un message contenant "Message :" et en extrait le texte via regex + `html2plaintext`.
- `_first_msg_dt(mod, rid)` : date du premier message (comment/notification non vide) sur l'enregistrement.
- Selon le `model` :
  - `nc_management.nonconformity` : expéditeur = `sent_by_id` ou `signale_par_id` ou `submitted_by_id` ou `create_uid` (si différent de l'utilisateur courant). Direction/service/department depuis la FNC. `send_datetime`/`message` via `_first_msg_dt`/`_wizard_note`.

  - `nc_management.smi_improvement_plan` : expéditeur = `submitted_by_id` ou `direction_id`. `send_datetime` = `date_soumission`, `message` = `description`.
- Toute exception est silencieusement ignorée (`except Exception: pass`), retourne le dict par défaut (champs vides).

## Synthèse des relations entre fichiers / modèles

### Hiérarchie SMI à 3 niveaux

```
Niveau 1 : PlanActionSmi (is_global=False)
    │  improvement_plan_id ──────► Niveau 2 : SmiImprovementPlan (par direction)
    │                                    │  global_plan_id ──────► Niveau 3 : smi_global_plan
    │  direct_global_plan_id ─────────────────────────────────────► Niveau 3 : smi_global_plan
    │
    └─ (ancien système) global_plan_id ──► PlanActionSmi (is_global=True)  [niveau 3 "legacy"]
```

- **Niveau 1** (`PlanActionSmi`, `is_global=False`) : plans d'action individuels créés par les directions. Référence `nature`, `fnc_id`, `responsable_id`/`responsable_ids`, `avancement`, `efficacite`, `state`.
- **Niveau 2** (`SmiImprovementPlan`, dans smi_improvement_plan.py) : regroupe les `plan_ids` (One2many vers niveau 1 via `improvement_plan_id`) d'une direction. Calcule des statistiques agrégées (`_compute_stats`). Soumission via `action_soumettre` qui :
  - Marque `state='soumis'`.
  - Crée/rattache au `smi_global_plan` unique (niveau 3, modèle séparé `nc_management.smi_global_plan` — non présent dans les deux fichiers analysés, probablement dans un autre fichier du module).
  - Auto-consolide les `plan_ids` dans le plan RMQSE actif (`PlanActionSmi.is_global=True`), via `global_plan_id` (ancien système) + `submission_state='integre'`.
- **Niveau 3 legacy** (`PlanActionSmi`, `is_global=True`) : agrège les plans individuels via `child_plan_ids` (One2many sur `global_plan_id`). Fournit les statistiques globales (`_compute_global_stats`), l'analyse d'efficacité (`_compute_analyse_html`, `_get_analyse_rows`), l'historique (`_compute_plan_historique_html`), et le cycle de clôture/recréation (`action_cloturer_plan`).
- **Notification croisée** : quand un plan de niveau 1 rattaché à un `SmiImprovementPlan` déjà soumis est modifié (champs dans `_GLOBAL_TRACKED_FIELDS`) ou créé après coup, une notification est postée dans le chatter du `global_plan_id` (niveau 3) correspondant — implémenté dans `PlanActionSmi.write()` et `PlanActionSmi.create()`.

### Flux FNC → FAC (Nonconformity ↔ CorrectiveAction)

- `Nonconformity.fac_ids` (One2many) ↔ `CorrectiveAction.fnc_id` (Many2one, `ondelete='set null'`).
- `Nonconformity.write()` :
  - Synchronise `description`/`analyse_causes`/`direction_id` vers les FAC liées (sync "intelligente", n'écrase pas si modification manuelle).
  - Crée automatiquement une FAC quand le traitement de la FNC est complet (tous les champs requis renseignés, `state` ∈ submitted/in_progress, `assigned_to_id` défini).
  - Synchronise `responsable_id`/`responsable_analyse_id` sur les FAC quand `assigned_to_id` change.
- `CorrectiveAction._onchange_fnc_id` : remplissage initial inverse (FNC → FAC) lors de la sélection manuelle d'une FNC.
- `CorrectiveAction.action_line_ids` (One2many) ↔ `ActionLine.fac_id` (Many2one, `ondelete='cascade'`). `ActionLine.direction_id` est un champ `related` sur `fac_id.direction_id` (stocké, readonly).

### Dashboard (NcDashboard)

Modèle sans données propres, consomme `Nonconformity`, `CorrectiveAction`, `PlanActionSmi`, `SmiImprovementPlan`, `hr.department`, `hr.employee`, `mail.message` pour produire les agrégats consommés par le frontend JS (3 tableaux de bord : RMQSE `get_stats`, utilisateur standard `get_user_stats`, drill-down direction `get_direction_details`, plus `get_plan_smi_stats`, `get_efficacite_categorie`, `get_sender_info`).

### Référentiels et héritages transverses

- `HrDepartmentScaek` (`scaek_level`) est le socle de filtrage hiérarchique utilisé par `Nonconformity`, `CorrectiveAction`, `PlanActionSmi`, `SmiImprovementPlan`, `ActionLine` (related), et `NcDashboard` (recherche des directions/départements/services).
- `NcType` est un référentiel autonome (catégories miroir des champs `type_*` de `Nonconformity`), sans lien de clé étrangère direct visible dans ces deux fichiers.
- `DocumentRevision` est indépendant fonctionnellement mais référence les trois types de documents (`fnc`, `fac`, `plan_smi`) via le champ `doc_type` et génère des liens PDF correspondants.
- `ResUsers` (héritage `res.users`) : propage la langue utilisateur vers son `partner_id` — utilitaire transverse sans lien direct avec la logique métier NC/FAC/SMI.
