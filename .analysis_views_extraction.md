**`view_nonconformity_form`** (`nc_management.nonconformity.form`)

Structure du formulaire :
- **Header** :
  - Bouton `action_open_number_wizard` "Générer Numéro FNC" (`oe_highlight`, visible seulement si `name == 'New'`)
  - Bouton `action_open_send_wizard` "Envoyer" (`oe_highlight`, invisible si `name == 'New'`)

- **Sheet** :
  - Titre `h1` = `name` (readonly)
  - Groupe **"En-tête"** :
    - Colonne 1 : cascade hiérarchique `direction_id` → `department_id` (domaine `scaek_level=departement`, `parent_id=direction_id`) → `service_id` (domaine `scaek_level=service`) → `section_id` (domaine `scaek_level=section`) → `equipe_id` (domaine `scaek_level=equipe`). Chaque champ devient `invisible` si son parent est vide, et `readonly` sauf en `draft`. Tous avec `options={'no_create': True, 'no_create_edit': True}`.
    - Colonne 2 : `date` (readonly sauf en `draft`)
  - Groupe **"Type de Non-Conformité"** : deux colonnes de checkboxes
    - Colonne 1 : `type_nc_produit`, `type_reclamation`, `type_sst`, `type_environnement`, `type_travaux`
    - Colonne 2 : `type_audit_interne`, `type_audit_externe`, `type_achat`, `type_reception`, `type_dysfonctionnement`, `type_autre` + `type_autre_preciser` (visible/`required` seulement si `type_autre=True`)
    - Tous readonly sauf en `draft`.
  - Groupe **"1- Description de la non-conformité"** :
    - `description` (textarea pleine largeur, placeholder, readonly sauf `draft`/`submitted`)
    - Sous-groupe `col="6"` : `signale_par_id`, `date_signalement`, `fonction_visa` (readonly sauf `draft`/`submitted`)
  - Groupe **"Traitement"** (`col="6"`) : checkboxes `trait_reprise`, `trait_declassement`, `trait_retour_fourn`, `trait_recyclage`, `trait_reparation`, `trait_autre` + `trait_autre_preciser` (visible/required si `trait_autre=True`). Readonly sauf `submitted`/`in_progress`.
  - Groupe **"2- Action immédiate"** :
    - `action_immediate` (textarea)
    - Sous-groupe : `realise_par_id`, `date_realisation` (readonly sauf `submitted`/`in_progress`)
  - Groupe **"3- Analyse des causes"** :
    - `analyse_causes` (textarea)
    - Label fixe en gras "Impact : coût, incidence, risque"
    - `impact` (textarea), readonly sauf `submitted`/`in_progress`
  - Groupe **"Fiche d'Action Corrective"** :
    - `can_access_fac` (invisible, champ technique)
    - `assigned_to_id` (readonly sauf `submitted`)
    - `fac_reference` "N° Fiche d'action" (readonly, visible si `can_access_fac=True`)
    - `fac_number_display` "N° Fiche d'action" (readonly, visible si `can_access_fac=False`)
  - Groupe **"Validation hiérarchique"** (`col="6"`) :
    - `superieur_id` (readonly sauf `in_progress`/`validated`)
    - `date_validation` (readonly sauf `in_progress`/`validated`)
    - `signature` (readonly sauf `in_progress`)

**Observation** : le champ `state` du FNC a 4 statuts visibles dans la statusbar (`draft, submitted, in_progress, validated`), aligné avec les `attrs` readonly conditionnés sur ces états.

## 12. Synthèse — observations transverses (toutes sections)

1. **Bug confirmé** : `wizard/export_smi_analysis_wizard.xml` — `<field name="target:">new</field>` contient un `:` parasite dans le nom du champ. À corriger en `<field name="target">new</field>`.

2. **Référence Plan SMI codée en dur** : `reports/report_plan_action_smi_pdf.xml` (template `report_plan_smi_content`) affiche "PACA-SMI 019/700", "Révision N° : 02", "Date de révision : 11/01/2024" en dur, alors que `reports/report_fnc.xml`/`report_fac.xml` recherchent dynamiquement la `document_revision` valable. Risque de désynchronisation si une nouvelle révision du Plan SMI est enregistrée dans `nc_management.document_revision`.

3. **Duplication FNC/FAC report templates** : `report_fnc_template` et `report_fac_template` partagent ~80% de code (styles, boucle sections/lignes). Candidat à factorisation (non bloquant).

4. **Duplication dashboard QWeb** : les 3 templates `NcDashboard`, `NcPlanSmi`, `NcUserDashboard` dans `static/src/xml/dashboard.xml` dupliquent intégralement le bloc "Calendrier" (`.cal-*`) et le bloc "Alertes & Notifications" (`<details>`). Candidat à factorisation via `t-call` de sous-templates communs.

5. **Actions/menus potentiellement orphelins** :
   - `action_nc_plan_smi` (tag `nc_plan_smi`, dans `views/dashboard.xml`) n'a pas de `<menuitem>` — ouvert dynamiquement (probablement depuis le bouton "Export" ou un autre menu/action serveur non couvert par cette analyse).
   - `action_smi_improvement_plan_rmqse` (dans `views/smi_improvement_plan_views.xml`) n'a pas de `<menuitem>`.
   - `view_plan_smi_graph_state`, `view_plan_smi_graph_direction`, `view_plan_smi_pivot` (dans `views/plan_action_smi.xml`) ne sont référencées par aucun `view_ids` d'action fenêtre dans ce fichier.

6. **Duplication d'action wizard "Générer Numéro"** : `action_fnc_number_wizard` (views/views.xml) et `action_number_generator_wizard` (wizard/number_generator_wizard.xml) pointent vers le même modèle `nc_management.number_generator_wizard` avec des libellés légèrement différents ("Générer Numéro FNC" vs "Générer Numéro"). À vérifier si les deux sont réellement utilisés.

7. **Incohérence `required` sur `date_consultation`** : `view_consulter_version_improvement_wizard_form` (wizard `consulter_version_improvement_wizard`) déclare `date_consultation` avec `required="1"`, alors que les vues équivalentes dans `wizard/consulter_version_wizard.xml` (`view_consulter_version_wizard_form` et `view_analyser_efficacite_wizard_form`) ne le déclarent pas `required`. Incohérence UX mineure (l'utilisateur pourrait valider sans date dans certains wizards et pas d'autres).

8. **Modèle `consulter_version_wizard` à double usage** : le même modèle Python sert deux vues UI différentes ("Consulter une version passée" et "Analyser l'efficacité à une date"), avec un champ additionnel `return_view_ref` uniquement dans la 2e. Cela fonctionne mais mélange deux responsabilités dans un seul modèle — à surveiller si le wizard évolue.

9. **Auteur placeholder** : `__manifest__.py` a `'author': 'Your Name'` — à personnaliser (SCAEK / nom réel de l'auteur) avant toute publication ou distribution externe du module.

10. **Cohérence menus "Dashboard"** : deux menus distincts nommés "Dashboard" (un pour `group_responsable_qualite` → `action_nc_dashboard`, un pour `group_nc_utilisateur` → `action_nc_user_dashboard`), même `sequence=1`. Pas de collision réelle car les groupes sont mutuellement exclusifs dans l'usage normal, mais à garder en tête si un utilisateur appartient aux deux groupes (il verrait potentiellement les deux menus "Dashboard").

12. **Pattern "Consulter version historique"** : présent à 3 niveaux (`plan_action_smi` via `consulter_version_wizard`, `smi_improvement_plan` via `consulter_version_improvement_wizard`, `smi_global_plan` via `date_consultation` directement sur le formulaire sans wizard dédié apparent). Le niveau 3 (`smi_global_plan`) semble utiliser un champ `date_consultation` + `vue_historique` directement sur le modèle plutôt qu'un wizard séparé — légère incohérence d'architecture entre niveaux (wizard vs champ direct), bien que fonctionnellement cohérente côté utilisateur (même bandeau "vue historique" partout).
